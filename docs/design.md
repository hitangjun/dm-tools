# DM数据库SQL优化分析工具 - 设计文档

## 1. 项目概述

### 1.1 背景

生产环境中有大量SQL执行缓慢，DBA和开发人员需要一个便捷的桌面工具来快速分析SQL性能瓶颈。本工具针对达梦(DM)数据库，提供执行计划分析、索引建议、统计信息检查和SQL写法规范检查四大核心能力，帮助用户在本地快速定位SQL性能问题并获取优化建议。

### 1.2 目标

- 提供一个开箱即用的桌面应用，无需部署Web服务
- 支持手动输入SQL进行即时分析
- 连接DM数据库获取执行计划和统计信息
- 不连接数据库时也能进行SQL写法检查和基础索引建议
- 以直观的评分和问题列表形式呈现分析结果

### 1.3 技术选型

| 项目 | 选择 | 理由 |
|------|------|------|
| 开发语言 | Python 3.10+ | 生态丰富，开发效率高 |
| GUI框架 | PySide6 (Qt6) | 跨平台，界面美观，功能强大 |
| DM数据库驱动 | dmPython | 达梦官方Python驱动 |
| SQL解析 | sqlparse | 纯Python SQL解析库 |
| 配置管理 | dataclasses + json | 轻量，无额外依赖 |

## 3. 核心模块设计

### 3.1 DM数据库连接器 (dm_connector.py)

**职责**: 封装dmPython的所有数据库操作，为上层提供统一的查询接口。

**核心方法**:

| 方法 | 功能 | 返回 |
|------|------|------|
| `connect()` | 连接DM数据库 | bool |
| `disconnect()` | 断开连接 | void |
| `execute(sql)` | 执行查询SQL | QueryResult |
| `get_explain_plan(sql)` | 获取SQL执行计划 | str (格式化文本) |
| `get_table_info(table)` | 获取表基本信息 | dict |
| `get_table_columns(table)` | 获取表列信息 | list[dict] |
| `get_table_indexes(table)` | 获取表索引信息 | list[dict] |
| `get_table_stats(table)` | 获取表完整统计信息 | dict |

**DM执行计划获取**: 通过 `EXPLAIN <SQL>` 语句获取，兼容部分版本的 `EXPLAIN FOR <SQL>` 语法。

**元数据查询**: 使用DM的 `ALL_TABLES`、`ALL_TAB_COLUMNS`、`ALL_INDEXES`、`ALL_IND_COLUMNS` 系统视图，SQL语法与Oracle兼容。

### 3.2 执行计划分析器 (plan_analyzer.py)

**职责**: 解析执行计划文本，识别性能问题。

**检测规则**:

| 问题类别 | 检测模式 | 风险等级 | 说明 |
|---------|---------|---------|------|
| 全表扫描 | TABLE SCAN FULL / SSCN | CRITICAL | 大表全表扫描是首要性能杀手 |
| 笛卡尔积 | CARTESIAN / CROSS JOIN | CRITICAL | 产生大量中间结果集 |
| 嵌套循环 | NESTED LOOP / NLJOIN | WARNING | 驱动表大时效率差 |
| 排序操作 | SORT / ORDER BY | INFO | 大数据量排序消耗内存 |
| 高代价操作 | HASH JOIN / HASH GROUP | INFO | 内存消耗较大的操作 |

**代价评分**: 基于问题数量和严重程度计算0-100分，越高越好。评分公式:

```
score = 100 - (CRITICAL × 15 + WARNING × 8 + INFO × 3 + 全表扫描次数 × 5)
```

### 3.3 索引建议器 (index_advisor.py)

**职责**: 解析SQL语句，识别WHERE条件、JOIN连接、ORDER BY、GROUP BY中使用的列，给出索引建议。

**分析流程**:

1. 使用sqlparse解析SQL语法树
2. 提取FROM/JOIN子句中的表名和别名
3. 提取WHERE条件中的列名和操作符(等值/范围)
4. 提取JOIN ON条件中的连接列
5. 提取ORDER BY / GROUP BY中的列
6. 结合已有索引信息，生成索引建议DDL
7. 检查是否与已有索引冲突(完全覆盖/前缀匹配)

**优先级**:

| 优先级 | 建议类型 | 原因 |
|--------|---------|------|
| 5 (最高) | JOIN连接列 | 连接列无索引导致嵌套循环低效 |
| 4 | WHERE等值条件 | 等值过滤是最常见的索引使用场景 |
| 3 | WHERE范围条件 | 范围查询可利用索引但效率略低 |
| 2 | ORDER BY / GROUP BY | 索引可避免排序操作 |

### 3.4 统计信息检查器 (stats_checker.py)

**职责**: 检查表和索引的统计信息是否完整和最新。

**检查项**:

| 检查项 | 条件 | 风险等级 | 说明 |
|--------|------|---------|------|
| 缺少统计信息 | LAST_ANALYZED为空 | CRITICAL | 优化器无法做准确决策 |
| 统计信息过期 | 超过7天未更新 | WARNING | 数据可能已大幅变化 |
| 行数为0 | NUM_ROWS=0 | WARNING | 可能统计信息未更新 |
| 列缺少NDV | NUM_DISTINCT为空 | INFO | 列级统计不完整 |
| 索引缺少统计 | 索引LAST_ANALYZED为空 | WARNING | 索引统计缺失 |

**建议命令**: 对于缺失或过期的统计信息，建议执行:
```sql
-- 收集表统计信息
DBMS_STATS.GATHER_TABLE_STATS('表名');
-- 或
ANALYZE TABLE 表名 COMPUTE STATISTICS;

-- 收集索引统计信息
DBMS_STATS.GATHER_INDEX_STATS('表名', '索引名');
```

### 3.5 SQL写法规范检查器 (sql_linter.py)

**职责**: 检查SQL中的常见写法问题，这些问题可能导致索引失效或性能下降。

**检查规则**:

| 规则ID | 规则名称 | 等级 | 问题 | 建议 |
|--------|---------|------|------|------|
| R001 | SELECT * | WARNING | 返回所有列，无法利用覆盖索引 | 明确列出需要的列 |
| R002 | 隐式类型转换 | WARNING | 字符串与数字比较导致索引失效 | 确保类型一致 |
| R003 | OR条件 | INFO | 可能无法使用索引 | 改写为UNION ALL |
| R004 | LIKE前导通配符 | WARNING | '%keyword'必定全表扫描 | 避免前导% |
| R005 | 函数作用于列 | WARNING | WHERE中UPPER(col)导致索引失效 | 改写或建函数索引 |
| R006 | 未限制结果集 | INFO | 无TOP/LIMIT可能返回大量数据 | 添加TOP N |
| R007 | 大IN列表 | WARNING | >10个元素的IN列表效率差 | 使用临时表JOIN |
| R008 | WHERE子查询 | INFO | 子查询可能效率低 | 改写为JOIN |
| R009 | 不等于操作符 | INFO | != / <>通常无法用索引 | 改写为范围查询 |
| R010 | IS NOT NULL | INFO | 通常无法有效使用索引 | 考虑过滤索引 |

## 5. 关键设计决策

### 5.1 核心模块无UI依赖

所有 `core/` 下的模块不依赖PySide6，可以独立使用。这意味着:
- 可以在命令行脚本中直接调用分析模块
- 方便后续扩展为Web服务或CI/CD流水线工具
- 单元测试更容易编写

### 5.2 后台线程分析

SQL分析(特别是获取执行计划和统计信息)涉及网络IO，使用QThread在后台执行，避免界面冻结。AnalyzeWorker线程负责:
- 调用各核心模块
- 协调模块间的数据传递(如统计信息用于增强索引建议)
- 通过Signal将结果传回主线程

### 5.3 渐进式分析

分析按依赖关系分层执行:
1. **第一层(无需连接)**: SQL规范检查 + 基础索引建议 — 立即返回
2. **第二层(需要连接)**: 执行计划获取与分析
3. **第三层(需要连接)**: 表统计信息查询与检查 + 增强索引建议

即使数据库连接失败，用户也能获得第一层的分析结果。

### 5.4 DM数据库兼容性

DM数据库的SQL语法与Oracle高度兼容，本工具在设计时考虑了:
- 执行计划获取使用 `EXPLAIN` 语法(兼容DM7/DM8)
- 元数据查询使用 `ALL_*` 系统视图
- DM执行计划操作符(SSCN/SSEK/HJOIN/NLJOIN等)的模式匹配
- 统计信息收集建议使用DM的 `DBMS_STATS` 包

## 6. 安装与使用

### 6.1 环境要求

- Python 3.10+
- 达梦数据库客户端(dmPython依赖DM客户端库)
- PySide6 (Qt6)

### 6.2 安装步骤

```bash
# 1. 安装Python依赖
pip install -r requirements.txt

# 2. 安装DM客户端(如需要连接DM数据库)
#    从达梦官网下载DM客户端，安装后设置环境变量
#    或将dmPython的依赖库路径加入LD_LIBRARY_PATH

# 3. 运行
python main.py
```

### 6.3 使用流程

1. 在顶部"数据库连接"区域输入DM数据库的连接信息
2. 点击"连接"按钮建立数据库连接(可选，但推荐)
3. 在左侧SQL编辑器中输入或粘贴要分析的SQL语句
4. 点击"分析"按钮(或从下拉菜单选择示例SQL快速体验)
5. 在右侧结果面板查看分析结果:
   - "总览"页查看综合评分和关键问题
   - 各Tab页查看详细分析结果和建议
6. 如需保存报告，通过"文件 > 保存分析报告"导出

## 8. 附录

### 8.1 DM执行计划常见操作符

| 操作符 | 含义 | 性能关注点 |
|--------|------|-----------|
| SSCN | 全表扫描 | 大表上性能差 |
| SSEK | 索引等值扫描 | 性能好 |
| MSEK | 索引范围扫描 | 取决于范围大小 |
| HJOIN | 哈希连接 | 内存消耗大 |
| NLJOIN | 嵌套循环连接 | 驱动表大时差 |
| MJOIN | 归并连接 | 需要排序 |
| SORT | 排序 | 内存/临时空间 |
| HAGR | 哈希聚合 | 内存消耗 |
| PRJT | 投影 | 通常无问题 |
| SLCT | 选择过滤 | 配合索引使用 |

### 8.2 配置文件格式

```json
{
    "connection": {
        "host": "127.0.0.1",
        "port": 5236,
        "user": "SYSDBA",
        "password": "SYSDBA",
        "schema": "",
        "timeout": 30
    },
    "slow_threshold_ms": 1000,
    "max_rows_explain": 10000,
    "enable_auto_stats": true,
    "highlight_risk_level": "all"
}
```
